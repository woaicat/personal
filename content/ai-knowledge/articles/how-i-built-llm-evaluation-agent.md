大家都在谈 Agent，但很多项目讲到最后，仍然只是一个更长的 Prompt，外加一个聊天窗口。

对我来说，更实际的问题是：能不能把一份需求文档交给系统，让它不只给出几段建议，而是沿着一套固定流程，真正完成一整套大模型评测方案交付？

这是我做这个项目的起点。

在设计大模型产品评测时，我会反复做几件事：理解产品、拆解风险、定义指标、规划数据集、准备样本，再把这些内容整理成测试执行手册。每次面对的产品不同，但工作框架高度相似。直接使用通用对话模型当然可以完成其中一部分，可我仍然需要反复解释背景、提醒它衔接前文，还要自己检查产物有没有缺失。

所以我做了一个本地运行的“大模型评测方案设计智能体”。它不会替代真正的测试执行，也不是一个自动给模型打分的裁判。它负责的是评测开始之前最费脑力、也最容易遗漏的部分：把产品需求转化为一套可执行的评测方案和配套物料。

这篇文章会完整拆解它的工作流、架构、提示词设计、运行方式，以及我在做完这个项目之后，对 Agent 产品的一些新理解。

## 工作流：让 Agent 完成一次评测方案交付

这个项目的目标很明确：用户填写产品名称，上传 Markdown 或 Word 需求文档，再补充少量约束，系统就自动完成分析、规划、产物生成和质量检查。

如果使用通用聊天工具，我需要告诉它“先理解产品，再设计指标，然后生成样本，最后写执行手册”。下次换一个项目，还要把这些要求重新说一遍。

而在这个智能体里，任务边界、执行顺序、交付格式和验收标准都已经被写进系统。用户不需要学习一套复杂的提示词，只需要把项目材料交给它。

这也是我理解的专用 Agent：不是让模型什么都能做，而是让它对一个重复任务足够熟悉。

### 第一步：先把系统拆成几个清晰的部分

在写具体流程之前，我先确定了五个核心层次：

- **交互界面**：一个原生 HTML、CSS 和 JavaScript 页面，用来提交需求、查看计划、跟踪进度和下载文件。
- **API 服务**：FastAPI 负责接收表单与文档，创建任务，并通过 SSE 把实时事件推送给前端。
- **Agent 工作流**：LangGraph 把规划、执行、工具调用、质检、反思和总结连接成显式状态机。
- **上下文层**：需求文档、表单信息、历史步骤结果和可选的向量检索共同组成每一步的执行上下文。
- **产物层**：每次运行都会建立独立目录，保存 Word、CSV、脚本、清单和上下文快照。

整个过程可以简化成：

```text
需求文档与产品信息
        ↓
Planner 生成行动计划
        ↓
Executor 准备当前步骤
        ↓
Tool 生成并保存真实文件
        ↓
Reflection QA 按验收标准质检
        ↓
继续 / 重做 / 追加补救步骤
        ↓
Summary 汇总结果与产物
```

这里最重要的不是用了多少技术，而是每一层都只负责一件事。模型负责理解和生成，工作流负责控制顺序，工具负责把结果落成文件，前端负责让人看见发生了什么。

### 第二步：把评测经验写成一张可执行的图

我没有让 Agent 自由决定所有事情，而是先固定了评测方案设计中最稳定的四个业务步骤：

1. **产品解析**：梳理业务目标、用户、核心流程、风险和评测关注点。
2. **评测方案设计**：定义评测范围、指标、方法、数据原则、执行流程和角色分工。
3. **数据与样本准备**：生成数据集规范和样本 CSV，明确字段、样本量、标注与复核要求。
4. **测试执行与物料**：输出执行手册，并根据评测方法派生提示词、自动化脚本或人工记录模板。

Planner 不是简单列一个待办清单。它必须为每一步写清楚预期产物、成功标准、前置检查和风险。如果输入信息不足，也不能悄悄补全，而要明确标记“假设”或“待补充”。

随后，LangGraph 按照 `Planner → Executor → Tool → Reflection QA → Reflection → Summary` 推进任务。每完成一步，质检节点都会将实际结果与验收标准比较，并做出三种决定：继续、重做，或者插入额外的补救步骤。

这让我获得了一个很关键的能力：流程有主干，但不是僵硬脚本。它既知道通常应该怎么做，也能在发现材料缺失时调整后续行动。

### 第三步：给智能体一个足够具体的角色

通用助手的系统提示词通常强调“有帮助”和“回答问题”，但这远远不够。我的系统提示词先给它设定了一个更具体的身份：

> 大模型评测方案总架构师 + 测试交付经理

这两个角色缺一不可。“架构师”要求它理解产品、评测指标和方法选择；“交付经理”则要求它关心文件是否齐全、谁来执行、怎样复核、出现问题后怎么办。

提示词中还写入了几条硬约束：

- 指标必须可以量化或核验。
- 数据集必须说明样本量、字段约束、标注和复核机制。
- 每份产物都要写清适用人群、注意事项和质量保障。
- 自动化脚本与 LLM 提示词不是为了展示能力，只有评测方法需要时才生成。
- 信息不足时必须显式写出假设、责任人或验证方式。

对于计划、样本表和反思结果，我要求模型返回固定 JSON 结构；如果格式不合法，系统会尝试提取有效片段，必要时再调用专门的修复提示词。

这部分看起来不如“自主规划”酷，但它决定了 Agent 的输出能不能被下一个节点稳定使用。对一个要连续工作几十步的系统来说，结构化契约往往比一段漂亮回答更重要。

### 第四步：让后续步骤真的记得前面做过什么

多步骤 Agent 很容易出现一个问题：每一步单看都像对的，合在一起却互相矛盾。

为了减少这种断裂，我把上下文分成了几类：用户填写的产品信息、文档解析结果、前序步骤生成的关键内容，以及最近一次执行产生的 insight。执行新步骤前，Context Manager 会把这些信息重新组装；如果配置了嵌入服务，还会从 Chroma 中检索与当前任务最相关的三条内容。

向量检索在这里是增强项，不是系统运行的前提。没有配置嵌入模型时，工作流仍然可以继续，只是不会获得额外的语义检索结果。

我很喜欢这个降级方式。一个真正可用的 Agent 不应该因为某个增强能力不可用，就让整条主流程直接崩溃。

### 第五步：把运行过程暴露给用户

用户提交任务后，前端会先显示 Planner 生成的步骤，再通过 SSE 持续接收执行事件。页面上可以看到当前步骤、尝试次数、工具调用、知识命中、反思结论和已经生成的文件。

如果网络中断，前端会自动重连并回放已经缓存的事件；用户也可以暂停，再从原有进度继续查看。后端使用 SQLite checkpointer 保存 LangGraph 状态，因此流程不只是存在于一段临时对话里。

这件事很重要。长任务最让人焦虑的并不是等待，而是不知道系统还在做什么、做到哪里、失败后能不能恢复。

我没有把 Agent 的思考包装成一个旋转的加载图标，而是把“计划—执行—质检—产物”变成用户可以理解的进度。

### 第六步：最终交付的不是聊天记录，而是一组文件

一次任务完成后，系统会根据实际方法生成一组可下载、可预览的产物，包括：

- 产品解析文档。
- 完整评测方案。
- 数据集准备规范。
- 一张或多张样本 CSV。
- 测试执行指导手册。
- LLM 评测提示词、自动化脚本或人工记录模板。
- 记录全部产物信息的 `manifest.json`。

文档内容在生成时使用 Markdown 结构，保存时会转换为 Word；CSV、脚本和模板则按各自格式落盘。每次运行都有独立目录，同时保存源文档副本和上下文快照，方便之后追溯这份方案是基于什么信息生成的。

这正是我认为这个项目最有价值的地方：它的结果不是“AI 好像分析过”，而是一套可以继续修改、评审和交接的工作底稿。

## 做完之后，我对 Agent 有了哪些新理解

### 1. 专用 Agent 的价值来自偏见，而不是万能

这里的“偏见”是指系统对正确工作方式有明确判断。它知道评测方案应该先理解产品，知道样本表必须依赖指标与场景，也知道信息不足时不能假装确定。

通用工具把选择留给每一次对话；专用 Agent 则把稳定的经验固化进流程。任务越重复，这种固化越有价值。

### 2. 文件比回答更接近真实工作

在真实团队里，人们很少需要一段无法复用的长回复。大家需要的是能评审的方案、能填写的表格、能执行的手册，以及能追踪的任务记录。

当我把“生成文件”而不是“回答问题”设为最终目标后，很多产品决策都变得更清楚：必须有文件清单，必须保留上下文，必须支持预览下载，也必须让不同产物之间保持一致。

### 3. 反思不是让模型再想一遍

如果没有明确标准，“请检查一下你的回答”通常只会得到另一段听起来合理的话。

我的 Reflection QA 会读取当前步骤的成功标准，检查章节、数据和派生产物是否齐全，再决定继续、重做或追加步骤。反思之所以有用，不是因为模型突然更聪明，而是因为它有了一个可以对照的检查表。

### 4. 可恢复和可观察，本身就是 Agent 能力

一个长流程系统不仅要生成正确结果，还要处理断线、重试、历史回放和状态恢复。SSE、任务事件、checkpoint、manifest 看起来都不是“智能”，却决定了用户敢不敢把真实工作交给它。

## 它现在还不是什么

这个项目目前仍是一个本地单用户基础版。我刻意没有先做多租户、复杂权限、云端数据库和任务队列，也没有把它包装成一套全自动评测平台。

它能帮助人完成评测设计与准备，但不能替代业务专家确认事实，也不能替代后续的真实测试执行。需求文档质量、模型稳定性和样本代表性，仍然会直接影响结果。长任务的恢复体验、更多工具节点和更完整的自动化测试，也还有继续打磨的空间。

但我认为这正是一个合适的起点：先把一个高频、边界清楚的工作做成闭环，再决定是否扩大能力。

## 结语：从一件反复发生的工作开始

做完这个项目后，我越来越不把 Agent 理解成“一个会自己做任何事的模型”。它更像是一套围绕模型搭起来的工作系统：有明确输入，有固定流程，有有限工具，有质量检查，也有可以交给别人的产物。

如果你也想做一个自己的 Agent，可以先不考虑多智能体、长期记忆或复杂编排。先看一看自己的工作：有没有一件事，你每次都要补充相似的背景、执行相似的步骤，并期待相似的结果？

那很可能就是最适合被做成专用 Agent 的任务。

我的项目源码可以在 [GitHub](https://github.com/woaicat/-) 查看。它还不是最终答案，但已经让我看到：当经验被写进流程，AI 才真正从一次性的回答，变成可以反复使用的生产力工具。
